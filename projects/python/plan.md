# Техническое задание
## Проект: Telegram-бот для практики английского языка с проверкой ошибок

---

## 1. Общая информация

**Тип проекта:** учебный, командный (≈4 человека)  
**Цель:** разработать микросервисное приложение для практики письменного английского языка через Telegram-бота и веб-интерфейс.  
Приложение должно работать в Docker и запускаться одной командой `docker-compose up`.

---

## 2. Цели и задачи проекта

### 2.1. Цель

Создать интерактивного Telegram-бота, который ведёт диалог с пользователем на английском языке и:
- исправляет его сообщения,
- объясняет допущенные ошибки простыми словами.

Параллельно должна существовать веб-версия (демо), которая использует тот же backend.

### 2.2. Задачи

- Реализовать Telegram-бота для общения на английском.
- Реализовать backend-сервис, который:
  - получает сообщения от бота и веб-интерфейса,
  - сохраняет историю в БД,
  - вызывает AI-сервис для анализа текста.
- Реализовать AI-worker-сервис, который:
  - обращается к LLM API,
  - возвращает исправленный текст и объяснение ошибок.
- Реализовать веб-интерфейс для демонстрации работы системы.
- Обернуть все сервисы в Docker-контейнеры и настроить запуск через `docker-compose`.

---

## 3. Целевая аудитория

- Студенты и учащиеся, изучающие английский язык.
- Пользователи, которые хотят практиковать письменный английский в формате чата.
- Преподаватели, которым нужен простой инструмент для тренировки и демонстрации.

---

## 4. Функциональные требования

### 4.1. Telegram-бот (`telegram_bot_service`)

Бот должен уметь:

1. Обрабатывать команды:
   - `/start` — приветствие, краткое описание бота и инструкция по использованию.
   - `/help` — подсказка по возможностям бота.
   - (опционально) `/level` — выбор уровня английского (A2/B1/B2).

2. Принимать произвольные текстовые сообщения на английском.

3. Для каждого сообщения:
   - отправлять текст в `backend_api_service` по HTTP (асинхронно),
   - получать ответ с исправленным текстом и объяснением ошибок,
   - отправлять пользователю форматированный ответ, например:
     - **Correct version:** `...`
     - **Explanation:** `...`.

4. Обрабатывать ошибки:
   - при недоступности backend’а или AI-сервиса выдавать понятное сообщение пользователю
     (например: "Service is temporarily unavailable, please try again later").

Бот **не должен** напрямую взаимодействовать с базой данных и LLM API.

---

### 4.2. Веб-интерфейс (`web_app_service`)

Веб-приложение должно:

1. Предоставлять форму для ввода текста на английском.
2. По кнопке "Проверить" отправлять запрос в `backend_api_service` (тот же endpoint, что использует бот).
3. Отображать:
   - исходный текст пользователя,
   - исправленный текст,
   - объяснение ошибок.
4. (Опционально) отображать историю последних N сообщений для текущей сессии.

---

### 4.3. Backend API (`backend_api_service`)

Backend должен:

1. Предоставлять HTTP API (REST, FastAPI), в том числе endpoint, например:
   - `POST /check` — проверка текста.

2. Логика `POST /check`:
   - принять от клиента (бот/веб):
     - идентификатор пользователя (например, `telegram_id` или session id),
     - текст сообщения,
     - (опционально) уровень английского;
   - валидировать данные (длина текста, пустой текст и т.п.);
   - сохранить/обновить информацию о пользователе в БД;
   - сохранить исходное сообщение в БД;
   - асинхронно вызвать `ai_worker_service` (`POST /analyze`) с текстом;
   - получить `corrected_text` и `explanation` от AI-сервиса;
   - сохранить результат в БД;
   - вернуть клиенту (боту или вебу) JSON с результатом.

3. Работать асинхронно (FastAPI + async HTTP-клиент).

---

### 4.4. AI worker (`ai_worker_service`)

Сервис должен:

1. Предоставлять внутренний endpoint, например:
   - `POST /analyze` — анализ текста.

2. Принимать JSON вида:

```json
{
  "text": "I has a dog.",
  "level": "B1",
  "history": [
    { "role": "user", "text": "Hi, my name is Alex." },
    { "role": "assistant", "text": "Nice to meet you!" }
  ]
}
