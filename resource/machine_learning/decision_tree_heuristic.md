## Pre-pruning
- В pre-pruning обычно входят различные критерии остановы, например
	- *Максимальная глубина дерева(max_depth)*
		- Жёсткий потолок. Самый простой способ контролировать сложность.
	- *Минимальное количество объектов в листе(min_samples_leaf)*
		- Сильнее, чем min_samples_split: проверяет обоих потомков, а не только родителя.
	- *Максимальное количество листьев(max_leaf_nodes)*
		- Количество конечных узлов дерева
		- Меняет стратегию роста на best-first: сначала самые важные разрезы.
	- *Минимальное количество разделений(min_samples_split)*
		- Защищает от дробления на статистически незначимых подвыборках.
	- *Минимальная информативность(min_impurity_decreas)*
		- Не разделяет узел, если критерий информативности не более порогового значения
		- Прямая проверка полезности разреза. Слабее pruning — не видит вперёд.
## Post-prining
- Идея противоположная предыдущей. Составляем дерево до конца, после чего объединяем узлы, которые не улучшают обобщение
### Reduced Error Pruning (REP)

- Постредукция просматривает все внутренние вершины дерева и заменяет их листом, если это не ухудшает качество на **контрольной выборке**. Процесс продолжается, пока есть узлы, удовлетворяющие критерию замены. Стандартная рекомендация — оставлять в контроле около 30% объектов.
### Cost-complexity pruning(CCP)
- Более изощренный метод, используемый в CART и sklearn. Вводится регуляризованный функционал ошибки, который штрафует за число листьев
- $$R_{\alpha}(T) = R(T) + \alpha \cdot |T|$$
	- Где $R(T)$ - *ошибка дерева*, $\alpha \geq 0$ - *параметр сложности*, а $|T|$ - *количество листьев дерева*
	- При $\alpha = 0$ получаем полное дерево, при $\alpha \to \infty$ получаем только корень
## Категориальные признаки
### One-hot encoding
- Кодирует классовый признак $c = {c_{1}, c_{2}, \dots, c_{K}}$ K новыми признаками
- Из за этого появляется большое количество признаков из одного
- *Плюс* - довольно просто
- *Минус* - создает много новых признаков
### Target encoding
- Заменяет признак на среднее значение таргета
- *Плюс* - Работает при большом K довольно неплохо
- *Минус* - Появляется утечка данных, из за которой необходимо использовать cross валидацию
### Native
- **CatBoost** и **LightGBM** могут перебирать подмножества с доп эвристиками, но это выходит *дороже вычислительно*
## Пропуски
### Взвешенное распределение
- Распределяет NaN значение в обе ветки, после чего ответ выдается как среднее между двумя ветками
### Суррогатные распределения
- Для каждого сплита по признаку j запоминаем «запасной» признак k, который даёт похожее разбиение
### Отдельная ветка NaN 
- NaN идет в *отдельную ветку* в **LigthGBM** и **XGBoost**, за счет чего получается учитывать NaN значения автоматически
